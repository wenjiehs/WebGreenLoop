# Sprint 执行协议

## 核心原则

**审批通过 = 授权连续执行。不要中途停下问"要继续吗"。**

## 执行规则

1. **连续执行** — 计划审批通过后，按优先级顺序连续执行全部任务，不等用户确认
2. **自动提交** — 每完成一个 Phase 做一次 `git commit + push`
3. **自动打勾** — 每个任务完成后在计划文档中标 ✅，失败的标 ❌ 并记录原因
4. **自行修 Bug** — 编译错误自行修复（最多重试 3 次，3 次不过才报告）
5. **最终汇报** — 全部完成后一次性汇报：完成了什么 / 跳过了什么 / 遇到的问题

## 什么时候才停下来问用户

只在以下情况暂停：
- 方向性决策（如"要不要删掉某个系统"）
- 连续 3 次编译失败无法自行解决
- 发现计划中的任务描述有矛盾或冲突
- 需要用户提供信息（如账号密码、外部 API key）

## Sprint 计划模板

创建新计划时，使用以下结构：

```markdown
# Sprint 计划 — [主题]

> 状态: **待审批**

## ⚡ 执行协议
（引用本规则）

## Phase 1: [名称]
### 1.1 [任务名]
- 当前问题: ...
- 修改内容: [ ] ...
- 验收标准: ...

## 📋 优先级表
| 优先级 | 任务 | 工作量 | 状态 |
|---|---|---|---|
| P0 | ... | 中 | ⏳ |

## ✅ 审批区
```

## 审批方式

用选择题让用户快速决策：
- Phase 1: ✅做 / 🔧部分做 / ❌不做
- Phase 2: ✅做 / 🔧部分做 / ❌不做
- 执行顺序: 你来安排 / 先做X再做Y
